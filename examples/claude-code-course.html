The entry point — where everything starts

The first file that runs is src/entrypoints/cli.tsx. It's surprisingly small — just 320 lines. Most of it is polyfills (fake implementations to replace build-time features).

CODE 
// Runtime polyfill for bun:bundle
const feature = (_name) => false;

// Simulate build-time macros
globalThis.MACRO = {
  VERSION: "2.1.888",
  BUILD_TIME: new Date().toISOString()
};
PLAIN ENGLISH

This fakes the build-time system. In the real Claude Code, these values come from the bundler. Here, we just hardcode them.

feature() always returns false, which disables experimental features.

MACRO holds version info and build timestamps.

The heart of the conversation — query.ts

query.ts (1,700+ lines) is where the magic happens. It manages the back-and-forth with the AI, decides when to call tools, and handles the streaming response.

CODE 
export async function query(
  messages: Message[],
  context: QueryContext
): Promise<void> {
  // Build request to send to API
  const response = await callClaudeAPI(
    normalizedMessages, params
  );

  // Process tool calls if AI asks for them
  for (const tool of toolCalls) {
    await executeTool(tool);
  }
PLAIN ENGLISH

This function takes your messages and context, then asks the AI for a response.

It sends everything to Claude's API and waits for a reply.

If the AI decides it needs to run a tool (like reading a file), it loops through and executes each one.

After tools run, their results go back to the AI for another response.

The Loop

Query.ts runs in a loop: send message → get response → if tools, run them → send results back → repeat until AI doesn't need more tools. This is called the tool-use loop.

PLAIN ENGLISH

The Clever Tricks

How Claude Code handles complexity gracefully

The tool system — giving AI hands

Tools are how Claude Code interacts with your files and system. There are 25+ built-in tools, each with a specific purpose:

BashTool
Run shell commands in a sandbox
FileReadTool
Read files, PDFs, images
FileEditTool
Edit files with string replacement
GrepTool
Search file contents
GlobTool
Find files by name pattern
AgentTool
Spawn sub-agents for parallel work

How tools are defined

Every tool follows the same pattern defined in Tool.ts:

CODE 
export type Tool = {
  name: string;
  description: () => Promise<string>;
  inputSchema: ZodType;
  call: (args, context) => Promise;
  isEnabled: () => boolean;
  isReadOnly: (input) => boolean;
};
PLAIN ENGLISH

name is what the AI uses to call the tool.

description tells the AI what the tool does.

inputSchema defines what arguments the tool needs (using Zod validation).

call is the actual implementation — what happens when the tool runs.

isEnabled controls if the tool is available.

isReadOnly determines if it modifies files.

The permission system — safety first

Before any tool runs, Claude Code checks if you approve. There are three permission modes:

🚫
Manual
Every tool use requires your approval
⚡
Auto
Approves automatically after first use
🔥
YOLO
Approves everything (dangerous!)
Why this matters

The permission system is 6,300+ lines of code — nearly as much as the query loop! It includes path validation, dangerous command detection, and rule matching. This is where Claude Code protects your system from accidental damage.

Context compaction — handling long conversations

Claude has a limit on how many tokens (words) it can process. When conversations get too long, Claude Code uses "compaction" to shrink the history.

Aha!

Token is a unit of text — roughly 4 characters = 1 token. Claude's context window is measured in tokens (e.g., 200K tokens = ~800K characters). When you exceed this limit, you get an error.

Claude Code has three compaction strategies:

  • Auto-compact: Runs in background, keeps conversation under limit
  • Micro-compact: Quick cleanup of redundant tool results
  • API compact: Uses Claude's built-in summary feature
